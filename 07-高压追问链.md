## 一、项目定位与边界

## 1. 这个项目到底解决什么业务问题？

### 主问题

这个项目到底解决什么业务问题？

### 一追

你说的是公开 FAQ 问答，那它和普通客服机器人本质差别是什么？

### 二追

如果业务目标只是答 FAQ，为什么值得单独做一套 RAG 系统？

### 三追

你怎么证明这个问题是“真实业务问题”，而不是你自己造出来的技术题？

### 面试官真正想考什么

- 你是否能从业务目标出发定义项目
- 你是否知道这个系统不是“为了上模型而上模型”
- 你是否能把问题说成一个值得投入资源解决的问题

### 你该怎么起手答

先一句话收口：  
**这个项目解决的不是“所有客服问题”，而是“公开 FAQ 如何被稳定命中、稳定回答、并且不过界”这个子问题。**

然后立刻补两层：

1. 用户问法不标准，FAQ 标题和真实 query 有表达鸿沟
2. FAQ 只能答公开规则，不能答订单、物流、退款状态这类个体问题

### 被继续追时怎么接

- 如果问“和普通客服机器人差别是什么”，就答：普通客服机器人偏对话自然度，这个系统偏证据约束和边界控制。
- 如果问“为什么值得单独做”，就答：因为高频问题答案本来就存在，关键在于命中率和风险控制，这正是 FAQ RAG 的价值。
- 如果问“怎么证明是业务问题”，就答：看重复咨询量、规则解释成本、FAQ 页面存在本身就说明这类问题是高频刚需。

### 最容易暴露的漏洞

- 一上来只讲架构，不讲场景
- 把问题说成“做一个智能助手”
- 没有讲清“自然语言命中率”和“不过界”这两个核心矛盾

## 2. 为什么它是 FAQ RAG，而不是通用 Agent？

### 主问题

为什么这个项目是 FAQ RAG，而不是通用 Agent？

### 一追

你都做了检索、生成、fallback，为什么这还不算 Agent？

### 二追

如果以后接一个工具查询接口，它是不是就变成 Agent 了？

### 三追

你现在这样定义，会不会只是因为项目不够复杂，所以不敢叫 Agent？

### 面试官真正想考什么

- 你是否真正理解 RAG 和 Agent 的能力边界
- 你是否会为了热点标签过度包装项目
- 你是否能稳住自己的定位，而不是被带偏

### 你该怎么起手答

先明确区分：

- **RAG 的核心**：证据召回、排序、受约束回答
- **Agent 的核心**：任务拆解、工具选择、状态推进、闭环执行

再落回本项目：  
这个系统不会主动规划“下一步做什么”，它只围绕 FAQ 做“找证据 -> 判断能否回答 -> 组织答案/拒答”。

### 被继续追时怎么接

- 如果问“为什么 retrieval + generation 不算 Agent”，就答：它有 LLM，不等于有 agentic 行为，关键看是否存在任务规划和执行闭环。
- 如果问“接一个工具是不是就变 Agent”，就答：不一定，单工具调用不等于 agent，关键还是有没有任务级决策和状态推进。
- 如果问“是不是不敢叫 Agent”，就答：不是不敢叫，而是项目边界就应该讲准，包装过度反而经不起深挖。

### 最容易暴露的漏洞

- 把“用了大模型”误说成“做了 Agent”
- 说不清 RAG 和 Agent 的分界
- 为了显得高级而硬贴 Agent 标签

## 3. 为什么只做公开 FAQ，不做订单、物流、退款状态查询？

### 主问题

为什么只做公开 FAQ，不做订单、物流、退款状态查询？

### 一追

这些问题用户也会问，为什么不一起做掉？

### 二追

如果接业务 API，不就能查了吗？为什么你现在一开始就排除？

### 三追

这样会不会让系统能力太弱，用户体验很差？

### 面试官真正想考什么

- 你是否能区分知识问答与状态查询
- 你是否有边界意识，而不是功能贪多
- 你是否能说清 trade-off

### 你该怎么起手答

先说结论：  
**公开 FAQ 问答和订单/物流/退款状态查询是两类完全不同的问题。**

再给三点区别：

1. 前者依赖公开知识，后者依赖实时私有数据
2. 前者是检索问题，后者是权限和事务问题
3. 前者答错偏体验问题，后者答错会直接上升到合规和信任问题

### 被继续追时怎么接

- 如果问“为什么不一起做”，就答：问题定义会混，系统风险也会陡增，不适合在 FAQ 样板阶段揉在一起。
- 如果问“接 API 不就行了”，就答：那已经不是 FAQ RAG 了，而是另一个带权限与状态查询的系统，需要单独设计。
- 如果问“会不会能力太弱”，就答：能力范围更窄是 trade-off，但换来的是系统可控性和更低越界风险。

### 最容易暴露的漏洞

- 回答成“以后可以做”，却没解释现在为什么不做
- 说不清 FAQ 与状态查询的本质差异
- 不敢承认能力范围收窄是代价

## 4. 你怎么定义“能答”和“不能答”？

### 主问题

你怎么定义“能答”和“不能答”？

### 一追

这个判断靠规则，还是靠模型？

### 二追

如果问题一半像 FAQ，一半像个体状态查询，你怎么判？

### 三追

“能答”和“检索到了东西”是一个概念吗？

### 面试官真正想考什么

- 你是否有明确的回答准入标准
- 你是否把“答不答”做成系统决策，而不是感觉
- 你是否能处理边界 case

### 你该怎么起手答

先给标准三件套：

1. 是否属于公开 FAQ 范围
2. 是否检索到足够相关且可追溯的证据
3. 回答是否会跨出 source 支撑边界

三个条件同时满足才答，否则就 fallback。

### 被继续追时怎么接

- 如果问“规则还是模型”，就答：边界优先靠明确规则守住，模型可以辅助，但不能成为唯一裁判。
- 如果问“半规则半状态”，就答：优先按高风险一侧处理，宁可保守 fallback，不把一般规则包装成个体结论。
- 如果问“检索到东西是不是就能答”，就答：不是，相关只是必要条件，不是充分条件。

### 最容易暴露的漏洞

- 说成“检索到就答”
- 说不清边界 case 怎么处理
- 没有把 source 和边界放进判断条件

## 二、检索与排序

## 5. 为什么要做混合检索，而不是只做向量检索？

### 主问题

为什么要做混合检索，而不是只做向量检索？

### 一追

那你具体观察到哪些 query 类型，向量检索不够用？

### 二追

如果关键词检索这么重要，为什么不干脆只保留 BM25？

### 三追

你怎么证明混合检索带来的收益，大于系统复杂度和延迟成本？

### 面试官真正想考什么

- 你是否理解不同召回器的职责
- 你是否知道 query 分布不是单一类型
- 你是否能讲清楚效果与复杂度的 trade-off

### 你该怎么起手答

一句话先答：  
**FAQ 场景既有语义改写问题，也有关键词强约束问题，单路向量检索覆盖不全。**

然后马上举两类对比例子：

- 口语化 query：更依赖 dense
- 术语型短 query：更依赖 BM25

### 被继续追时怎么接

- 如果问“哪些 query 向量不够用”，举“运费券可以提现吗”“价格保护可以提现吗”“退款申请 vs 退款到账”。
- 如果问“那为什么不只保留 BM25”，就反打：BM25 对术语强，但对口语化、改写、错别字明显掉点。
- 如果问“怎么证明值”，就回到评测：分 query 类型看 Recall@K、MRR、最终回答准确率和延迟成本。

### 最容易暴露的漏洞

- 只会说“向量不够”
- 没有例子
- 只讲收益，不讲索引维护和延迟代价

## 6. 为什么要做 query rewrite 和同义词扩展？

### 主问题

为什么要做 query rewrite 和同义词扩展？

### 一追

这两个能力的边界是什么？是不是重复建设？

### 二追

如果 rewrite 把用户原意改坏了怎么办？

### 三追

为什么不完全依赖 embedding 去解决表达差异？

### 面试官真正想考什么

- 你是否理解用户语言和知识库语言的分布差异
- 你是否知道 rewrite / synonym 的职责不同
- 你是否意识到 rewrite 也会引入副作用

### 你该怎么起手答

先答核心动机：  
**不是知识库没答案，而是用户问法和 FAQ 文档表达方式经常对不上。**

再分工讲：

- 同义词扩展：解决稳定业务同义表达
- query rewrite：解决更自由的口语化、非标准表达

### 被继续追时怎么接

- 如果问“边界是什么”，就答：同义词偏规则化映射，rewrite 偏语义规整。
- 如果问“改坏了怎么办”，就答：这是 rewrite 最大风险，所以必须做轻量改写并用评测监控误召回。
- 如果问“为什么不全靠 embedding”，就答：embedding 能兜一部分，但在术语不齐和表达很散时，前处理仍然能稳定增益。

### 最容易暴露的漏洞

- 把 rewrite 说成越激进越好
- 说不清 synonym 和 rewrite 的区别
- 完全不承认 rewrite 会改坏 query

## 7. 为什么选 RRF，而不是直接分数加权？

### 主问题

为什么选 RRF，而不是直接分数加权？

### 一追

不同召回器的分数为什么不能直接比？

### 二追

RRF 不是把原始分数信息丢了吗？这会不会太粗？

### 三追

什么时候你会考虑不用 RRF？

### 面试官真正想考什么

- 你是否真正理解异构召回器打分不可比
- 你是否能讲清 RRF 的优缺点
- 你是否知道它是工程折中，而不是万能最优解

### 你该怎么起手答

先答一句：

**我选 RRF，核心是因为 keyword、dense、sparse 的原始分数尺度不统一，直接加权很容易被某一路天然数值优势误导。**

然后强调：RRF 用排名融合，更稳。

### 被继续追时怎么接

- 如果问“为什么不能直接比”，就答：打分机制不同，数值大小不具备同一语义。
- 如果问“丢分数是不是太粗”，就答：是 trade-off，损失细粒度强弱信息，换来跨路融合稳定性。
- 如果问“何时替换”，就答：当有更多标注数据、希望做 learned fusion 时可以替换。

### 最容易暴露的漏洞

- 只会背公式
- 不承认 RRF 有局限
- 讲不清为什么分数不可直接比较

## 8. reranker 在这里的核心价值是什么？

### 主问题

reranker 在这里的核心价值是什么？

### 一追

没有 reranker 会怎样？最直接的错误类型是什么？

### 二追

如果 reranker 把正确结果排低了，你怎么定位？

### 三追

既然 reranker 很重要，为什么不把更多预算放在 rerank，而不是多路召回？

### 面试官真正想考什么

- 你是否理解召回与排序的分工
- 你是否知道 rerank 的收益和成本
- 你是否能定位 rerank 引起的问题

### 你该怎么起手答

一句话：

**召回解决“别漏”，reranker 解决“别排错”。**

然后举语义很近但业务结论不同的 FAQ 例子，比如“退款申请流程”和“退款到账时间”。

### 被继续追时怎么接

- 如果问“没有 reranker 会怎样”，就答：top1 不稳定，泛相关 FAQ 容易抢位。
- 如果问“排错了怎么定位”，就答：看 rerank 前后排序差异，检查是 query-doc 配对问题还是候选集本身就错。
- 如果问“为什么不重压 rerank”，就答：没有好的候选，rerank 也无米下锅，召回和排序不能互相替代。

### 最容易暴露的漏洞

- 说成“reranker 就是更高级模型”
- 不知道 rerank 最怕“候选集本来就错”
- 只讲好处，不讲延迟成本

## 9. chunk 策略怎么定的？失败案例有哪些？

### 主问题

chunk 策略怎么定的？失败案例有哪些？

### 一追

为什么不直接按固定字数切？

### 二追

chunk 太大和太小，各自最典型的坏处是什么？

### 三追

你怎么证明当前 chunk 策略比别的更好？

### 面试官真正想考什么

- 你是否真的做过知识切分
- 你是否知道 chunk 是在平衡完整性与检索精度
- 你是否能拿失败案例说明问题

### 你该怎么起手答

先答原则：

**chunk 不是随便定个长度，而是在“别切坏知识”和“别召回太泛”之间找平衡。**

再明确：FAQ 场景优先按语义单元切，再叠加长度约束。

### 被继续追时怎么接

- 如果问“为什么不固定字数切”，就答：规则类文本前提、例外和限制经常被切断。
- 如果问“太大太小坏处”，就答：太小缺条件，太大太泛。
- 如果问“怎么证明”，就答：看 Recall@K、命中 chunk 的信息完整性、最终答案是否漏条件。

### 最容易暴露的漏洞

- 答成“我设了个 300 字”
- 没有失败案例
- 不知道 chunk 问题最后会如何传导到答案

## 10. 为什么既存 FAQ 主文档，又存 chunk？

### 主问题

为什么既存 FAQ 主文档，又存 chunk？

### 一追

只存 chunk 不行吗？

### 二追

只存整篇 FAQ 不行吗？

### 三追

chunk 命中后，你怎么回溯到主文档再组织答案？

### 面试官真正想考什么

- 你是否理解不同粒度数据的职责
- 你是否能讲清召回单元和答案单元的差异

### 你该怎么起手答

先说分工：

- chunk：细粒度召回
- FAQ 主文档：完整语义归并、来源归属、最终展示

- 把两者说成重复存储
- 说不清为什么最终回答不直接基于 chunk

## 三、回答约束与越界防护

## 11. 你怎么判断该回答还是该 fallback？

### 主问题

你怎么判断该回答还是该 fallback？

### 一追

你说的 `confidence` 是怎么来的？是模型给的还是系统算的？

### 二追

如果业务方说你现在太保守了，想减少 fallback，你怎么办？

### 三追

“没找到答案”和“没找到足够好的证据”是一个概念吗？

### 面试官真正想考什么

- 你是否真的把拒答做成系统策略
- 你是否知道保守与激进之间的 trade-off
- 你是否能区分无答案与证据不足

### 你该怎么起手答

先给判断三件套：

1. 是否有足够相关的证据
2. 证据是否在允许边界内
3. 来源是否可追溯，且能支撑答案

不同时满足，就 fallback。

### 被继续追时怎么接

- 如果问 `confidence`，就答：它更像系统层综合决策分，不是单个模型心里打的分。
- 如果问“想减少 fallback”，就答：先优化检索和边界判断，不会直接放低阈值去换回答率。
- 如果问“没答案 vs 证据不足”，就答：前者是知识库里可能本来就没有，后者是当前检索链路没把它稳定找出来。

### 最容易暴露的漏洞

- 把 fallback 讲成“模型不自信”
- 说不清 decision signals
- 只会调阈值，不会谈优化方向

## 12. 你怎么防止模型回答订单、物流、退款这类越界问题？

### 主问题

你怎么防止模型回答订单、物流、退款这类越界问题？

### 一追

如果用户问“退款一般多久到账”和“我这笔退款多久到账”，系统怎么区分？

### 二追

你这个越界识别是靠规则还是靠模型？

### 三追

如果以后真要接业务状态查询，你现在这套边界设计还成立吗？

### 面试官真正想考什么

- 你是否把边界做进系统，而不是提示词
- 你是否能处理规则问题与状态问题的细微差别
- 你是否知道未来扩展不是推翻边界，而是增加新能力层

### 你该怎么起手答

先明确：

**我防越界不是靠模型自觉，而是从意图识别、来源约束、回答策略三层前置堵住。**

### 被继续追时怎么接

- 如果问“规则 vs 状态怎么区分”，就答：一个是一般规则解释，一个是个体状态判断，后者必须 fallback。
- 如果问“规则还是模型”，就答：高风险边界优先靠显式规则兜住，模型辅助，不做唯一判官。
- 如果问“以后接业务查询怎么办”，就答：那是新增一个有权限和实时数据支撑的查询能力，不是把 FAQ 系统硬改得更大胆。

### 最容易暴露的漏洞

- 回答成“prompt 会提醒模型不要乱答”
- 说不清“规则说明”和“个体状态”的分界
- 没有承认这类问题属于高风险输入

## 13. 你怎么证明返回的答案真的被 source 支撑？

### 主问题

你怎么证明返回的答案真的被 source 支撑？

### 一追

“有 source”和“被 source 支撑”有什么区别？

### 二追

如果 source 讲的是一般规则，模型却生成成了用户当前结论，你怎么发现？

### 三追

你会怎么评测 groundedness？

### 面试官真正想考什么

- 你是否理解 groundedness 不是挂链接就完事
- 你是否有办法识别 answer overclaim
- 你是否知道如何把 source 支撑做成验证项

### 你该怎么起手答

先立住一句：

**source 不是展示字段，而是回答资格的一部分。**

然后讲两层：

1. 流程上：先检索出可靠 source，再允许生成
2. 结果上：看答案关键信息点能否在 source 中逐项对上

### 被继续追时怎么接

- 如果问“区别是什么”，就答：有 source 只说明挂了链接，被 source 支撑才说明结论真的来自这个链接。
- 如果问“一般规则变成个体结论”，就答：这正是典型 overclaim / overreach，要通过边界 case 和 groundedness 检查发现。
- 如果问“怎么评测”，就答：看 source completeness，再看关键结论是否可在 source 中定位，必要时用人工或 LLM-as-judge 辅助。

### 最容易暴露的漏洞

- 以为 source 非空就算 grounded
- 说不清 overclaim case
- 完全不懂 groundedness 怎么评测

## 18. 如何避免“自己出题自己验证”带来的偏差？

### 主问题

如何避免“自己出题自己验证”带来的偏差？

### 一追

如果你自己做的项目，最容易在哪些地方出评测偏差？

### 二追

真实日志和人工构造 case，哪个更重要？

### 三追

你怎么防止评测集越来越像“为当前版本量身定制”？

### 面试官真正想考什么

- 你是否意识到评测也会过拟合
- 你是否知道如何引入更真实的数据分布

### 你该怎么起手答

先承认问题：

**自己做系统时，最容易把评测集做成“自己熟悉的问题合集”，而不是线上分布的近似。**

然后给三招：

1. 引入真实 query 或真实 query 改写
2. 按失败模式和 query 类型分层
3. 让别人 review 一部分 case 和标注标准

### 被继续追时怎么接

- 如果问“最容易偏在哪”，就答：happy path 过多、边界 case 太少、题目措辞太像文档原文。

## 20. 如果要把这个项目升级成企业知识库产品，先补哪三块？

### 主问题

如果要把这个项目升级成企业知识库产品，先补哪三块？

### 一追

为什么不是先补前端、Agent 化或者更多模型能力？

### 二追

这三块里你觉得最先卡住上线的是哪一块？

### 三追

如果只能在一个季度内做一块，你会选哪一个，为什么？

### 面试官真正想考什么

- 你是否具备从样板到产品的演进判断力
- 你是否知道企业系统真正的门槛不在“再接一个模型”

### 你该怎么起手答

先给答案：

1. 权限与多租户
2. 评测与可观测性
3. 数据与发布链路工程化

然后一句话点明原因：  
这三块分别对应安全、验证和稳定演进，是企业产品的底座。

### 被继续追时怎么接

- 如果问“为什么不是前端或 Agent 化”，就答：那些提升的是外在能力和形态，但先不解决权限、验证、发布，系统根本活不久。

- 如果问“一个季度只能做一块”，就答：优先权限与多租户，因为这是最硬的安全边界。

### 最容易暴露的漏洞

- 只会答“多模态、Agent、微调”
- 不知道企业化首先是安全和治理问题
- 讲不出优先级排序理由

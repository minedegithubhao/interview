## 2. 企业招聘什么能力最重要

如果只看岗位描述里被反复要求的能力，企业最看重的不是单点模型知识，而是从“模型能力”到“工程系统”再到“业务落地”的完整闭环。

| 能力方向 | 出现次数 | 覆盖比例 |
| --- | --- | --- |
| Agent/工作流 | 67 | 83% |
| 性能优化 | 67 | 83% |
| RAG/知识库 | 62 | 77% |
| 应用落地 | 61 | 75% |
| 后端/API集成 | 53 | 65% |
| 部署运维 | 48 | 59% |
| Prompt工程 | 45 | 56% |
| 模型训练/微调 | 43 | 53% |
| 评测/Eval | 30 | 37% |
| 领域知识 | 17 | 21% |

### 解释

- `Agent/工作流`、`RAG/知识库` 和 `应用落地` 同时高频，说明企业已经不满足于做聊天 Demo，而是要做能执行任务的业务系统。
- `性能优化` 和 `部署运维` 高频，说明企业并不只关心“做出来”，更关心成本、延迟、稳定性和可运维性。
- `后端/API集成` 出现很多，表明企业想要的是能把 AI 接进现有系统的人，而不是孤立做原型的人。
- `评测/Eval` 已经出现 30 次，说明这类岗位开始进入“效果可量化”的阶段。

### 结论

企业最看重的能力链路可以概括为：

`Python/后端能力 -> RAG/知识库 -> Agent/工作流 -> 部署与性能优化 -> 业务落地与评测`

## 9. 面试准备怎么做

### 总策略

不要按零碎知识点准备，要按企业买单的完整链路准备：

`业务问题 -> 数据与知识库 -> RAG -> Agent -> 系统集成 -> 部署优化 -> 评测闭环`

### 必备准备模块

1. 做一个可演示的完整项目
- 至少包含：API 服务、知识库/RAG、Agent 或工具调用、日志/监控、简单前端或控制台。
- 能够讲清为什么这样设计，而不是只展示效果。

2. 把 RAG 讲到工程层
- 文档解析、chunk、embedding、检索、重排、混合检索、幻觉控制、评测。

3. 把 Agent 讲到可控性
- 工具调用、任务拆解、失败重试、降级、Memory 管理、trace 和观测。

4. 把系统能力补上
- FastAPI / Flask / Django / Spring Boot 至少熟一个。
- 会 Docker 部署，知道基本 Linux 运维，能解释并发、缓存、限流和接口设计。

5. 如果目标是高薪岗，要补一个溢价方向
- 方向 A：vLLM / LoRA / 推理优化 / 微调
- 方向 B：多模态 / PyTorch / C++ / 视觉与端侧
- 方向 C：行业项目落地 + 企业级系统集成

### 典型面试题

- 为什么这个场景适合 RAG，而不是长上下文硬塞？
- RAG 的 chunk、embedding、rerank 如何设计？
- Agent 为什么需要工具调用？什么时候不该上 Agent？
- 如何评估一个 RAG/Agent 系统是否真的变好？
- 如何控制模型调用成本与响应延迟？
- 如果线上出现幻觉、超时、召回不准，如何定位问题？
- 为什么选 Python/Java？如何与现有业务系统集成？

## 10. 30 天准备路线

### 第 1 周
- 选一个真实业务场景，用 Python + FastAPI 做一个最小可用服务。
- 接入基础模型、向量库和简单 RAG。

### 第 2 周
- 加入 Agent 或工具调用。
- 做日志、缓存、重试、降级。
- 准备一小套评测集。

### 第 3 周
- 优化 chunk、召回、重排。
- 测试延迟、吞吐、成本。
- 输出系统设计文档。

### 第 4 周
- 二选一：补 vLLM/LoRA，或补 多模态/PyTorch。
- 把项目写进简历，形成 STAR 式表达。
- 模拟面试，重点练设计取舍。

## 11. 最终建议

如果目标是尽快拿到面试机会，最值得优先投入的路线是：

`Python + RAG + Agent + FastAPI + Docker + Eval`

如果目标是冲更高薪资，最应该追加的能力是：

`推理优化 / 微调 / 多模态 / 底层工程能力`

对于这批岗位来说，企业真正买单的不是“懂大模型概念”，而是“能把大模型做成稳定、可评估、可优化、能落地的生产系统”。

# PlotPilot 创作与审核手册

更新日期：2026-04-21

本文面向“要用 PlotPilot 真正写好一本小说”的作者、主编或项目负责人。它不是技术启动说明，而是一套与 PlotPilot 创作流程对齐的使用和审核方法。

核心原则：

- 你负责方向、审美、取舍和关键设定。
- PlotPilot 负责规划草案、批量生成、记忆维护和连续推进。
- 不要把自动驾驶当成“一键写完”的按钮。把它当成一个持续工作的写作团队。

## 1. 总流程

推荐使用流程：

```text
准备创意
  -> 创建小说
  -> 生成并审核 Story Bible
  -> 生成并审核知识库
  -> 启动自动驾驶
  -> 审核宏观规划
  -> 审核幕级/章节规划
  -> 生成章节
  -> 审核正文
  -> 维护知识库、伏笔、人物状态
  -> 继续下一章/下一幕
  -> 阶段性总检
```

对应系统阶段：

| 系统阶段 | 作者动作 | 目标 |
| --- | --- | --- |
| 创建小说 | 输入高质量 premise | 让系统知道“这本书到底写什么”。 |
| Bible 生成 | 审世界观、人物、地点 | 固定长篇设定，避免后续漂移。 |
| `macro_planning` | 审宏观规划 | 确定部/卷/幕和主线升级。 |
| `paused_for_review` | 确认或修改 | 决定是否进入下一阶段。 |
| `act_planning` | 审幕级章节大纲 | 保证每章有功能，不水。 |
| `writing` | 看正文和流式输出 | 把控人设、节奏、钩子。 |
| `auditing` | 看章后质量信号 | 判断是否需要修写。 |
| 章后维护 | 修知识库、伏笔、人物状态 | 保持长篇连续性。 |

## 2. 准备创意

创建小说前，先准备一份 500 到 1500 字的创意设定。不要只写一句类型梗。

应包含：

| 项目 | 审核问题 |
| --- | --- |
| 主角 | 他是谁？最想要什么？最怕什么？有什么缺陷？ |
| 核心矛盾 | 故事最大的压力来自哪里？敌人、制度、环境还是内心？ |
| 类型承诺 | 读者为什么会继续看？爽点、悬疑点、情绪点是什么？ |
| 世界规则 | 这个世界最重要的运行规则是什么？ |
| 金手指/特殊能力 | 能做什么？不能做什么？代价是什么？ |
| 主线方向 | 前 30 章大概解决什么？第一卷大概解决什么？ |
| 风格 | 轻松、黑色幽默、热血、克制、爽文、群像等。 |
| 禁区 | 不要写什么？不要走哪些套路？ |

不建议：

```text
一个程序员穿越修仙世界，靠 AI 助手变强。
```

更适合：

```text
三十岁程序员陈枫猝死后投胎到修仙世界。他极度厌恶前世内卷，只想低风险长生，但修仙世界比互联网大厂更残酷：资源靠抢，宗门靠卷，天赋决定命运。他的外挂不是万能系统，而是前世熟悉的 AI 助手，只能基于已有信息推演功法、识别风险、优化路线，不能凭空给资源，也不能直接提升修为。故事核心冲突是：一个拒绝做牛马的人，如何在最卷的修仙世界里用工程化思维活得逍遥。
```

通过标准：

- 主角目标清楚。
- 世界压力清楚。
- 金手指边界清楚。
- 第一卷大方向清楚。
- 有至少一个区别于同类作品的识别点。

返工信号：

- 只有题材，没有冲突。
- 主角没有欲望，只是“被剧情推着走”。
- 金手指太万能。
- 没有明确读者爽点。
- 世界规则太空泛。

## 3. 创建小说

创建小说时重点填好：

| 字段 | 建议 |
| --- | --- |
| 标题 | 可以先临时，但最好能表达题材气质。 |
| 简介 / premise | 尽量放入完整创意设定。 |
| 类型 | 尽量准确，影响系统选择类型策略。 |
| 目标章节 | 先小后大。验证阶段建议 10 到 30 章。 |
| 每章字数 | 建议 2000 到 3000 字，便于控制节奏。 |

建议先做一轮“小规模试写”：

```text
目标 10 章
每章 2500 字
不开全自动
前 3 章重点审
```

通过前 3 章后，再扩展目标章节。

## 4. Story Bible 审核

Story Bible 是后续自动写作的设定根目录。这里的错误会被系统不断继承。

重点看：

```text
世界观设定
人物设定
地点设定
风格说明
人物关系
地点关系
```

### 4.1 世界观审核

检查项：

| 项目 | 审核问题 |
| --- | --- |
| 基础规则 | 修炼、权力、资源、社会秩序是否清楚？ |
| 类型一致性 | 是否符合你要写的类型？ |
| 冲突来源 | 世界本身是否能持续制造冲突？ |
| 金手指边界 | 是否没有被写成无代价万能工具？ |
| 约束 | 主角为什么不能立刻解决所有问题？ |

通过标准：

- 世界规则能支持至少一卷剧情。
- 主角能力有边界。
- 世界压力和主角目标之间有冲突。
- 没有与 premise 明显矛盾的设定。

返工信号：

- 世界只是一堆名词。
- 修炼体系、势力体系、资源体系互相无关。
- 金手指直接破坏所有难题。
- 设定里出现错误主角名、错误时代、错误类型。

### 4.2 人物审核

每个重要人物至少要有：

```text
目标
动机
和主角的关系
利益冲突
可变化空间
说话/行为特征
```

检查表：

| 项目 | 审核问题 |
| --- | --- |
| 主角 | 是否主动？是否有稳定欲望？ |
| 主要同伴 | 是否不只是工具人？ |
| 主要对手 | 是否有自己的目标，而不只是坏？ |
| 导师/贵人 | 是否不会替主角解决一切？ |
| 女性角色 | 是否有独立目标，而不只是奖励？ |
| 名字一致性 | 是否有陈枫/李枫这类漂移？ |

通过标准：

- 每个主要人物都能说清“他想要什么”。
- 至少 2 到 3 组关系天然有冲突。
- 反派或竞争者能持续施压。
- 主角不是被动旁观者。

返工信号：

- 人物目标全是“帮助主角”。
- 反派只会无脑挑衅。
- 人物关系太和平，没有戏剧张力。
- 名字、身份、境界前后不一致。

### 4.3 地点审核

地点不是地图装饰，而是剧情机器。

检查项：

| 项目 | 审核问题 |
| --- | --- |
| 功能 | 这个地点会发生什么类型的剧情？ |
| 风险 | 这里有什么危险或限制？ |
| 资源 | 这里能给主角什么机会？ |
| 关系 | 它和其它地点怎么连接？ |
| 阶段 | 它适合前期、中期还是后期使用？ |

通过标准：

- 重要地点都有剧情功能。
- 地点之间存在流动路线。
- 至少有几个地点能自然触发冲突、交易、试炼或秘密。

返工信号：

- 地点只是名字列表。
- 每个地点功能差不多。
- 没有低级区域到高级区域的成长路径。

## 5. 知识库审核

知识库和三元组会影响后续上下文召回。错误事实会污染后文。

重点检查：

```text
人物身份
人物关系
地点归属
势力关系
物品归属
重要事件结果
修炼境界
金手指能力边界
```

通过标准：

- 每条事实都清楚、具体、可复用。
- 重要设定已进入知识库。
- 没有明显错误关系。

返工信号：

- 错误三元组：例如“苏清寒 是 陈枫的父亲”。
- 过度抽象事实：例如“陈枫 很厉害”。
- 把猜测写成事实。
- 旧版本设定没有删除。

建议频率：

```text
创建 Bible 后检查一次
前 3 章后检查一次
每 5 到 10 章检查一次
每卷结束检查一次
```

## 6. 宏观规划审核

宏观规划决定整本书的骨架。系统进入 `paused_for_review` 时，先看结构树，不要立刻点确认。

你要审核：

```text
部/卷/幕结构
每幕标题
每幕目标
每幕冲突
每幕结尾钩子
主角成长曲线
主线是否偏离 premise
```

### 宏观规划检查表

| 项目 | 审核问题 |
| --- | --- |
| 开局 | 第一幕是否能快速建立主角、世界、冲突？ |
| 升级 | 每一幕压力是否比上一幕更大？ |
| 转折 | 是否有清楚的失败、代价、反转？ |
| 主线 | 是否围绕核心矛盾推进？ |
| 人物 | 重要人物是否有出场和变化空间？ |
| 世界 | Bible 的人物、地点、规则是否被用上？ |
| 钩子 | 每卷/每幕结尾是否有继续阅读动力？ |

通过标准：

- 每幕都有独立的叙事目标。
- 冲突逐步升级，不是平铺。
- 主角在每一幕都有选择和代价。
- 结尾留有强钩子。
- 没有离开最初卖点。

返工信号：

- 大纲像流水账。
- 每一幕都只是“修炼、历练、变强”。
- 中段没有明确危机。
- 过早解决核心矛盾。
- 主角长期被动。
- 重要人物没有作用。

确认方式：

- 前端工作台自动驾驶面板点击 `确认大纲，继续写作`。
- 或调用 `POST /api/v1/autopilot/{novel_id}/resume`。

## 7. 幕级规划审核

幕级规划会把当前幕拆成章节。这里要看“章节功能”，不是只看标题。

每章必须至少完成一种功能：

```text
推进事件
制造冲突
揭示信息
改变关系
埋设伏笔
回收伏笔
完成选择
付出代价
打开新问题
```

### 章节规划检查表

| 项目 | 审核问题 |
| --- | --- |
| 事件 | 这一章发生了什么具体事件？ |
| 冲突 | 主角遇到了什么阻力？ |
| 选择 | 主角做了什么决定？ |
| 变化 | 章末和章初相比，局面有什么不同？ |
| 钩子 | 章末是否有继续阅读动力？ |
| 节奏 | 是否连续多章功能重复？ |
| 位置 | 这章是否服务当前幕目标？ |

通过标准：

- 每章都能一句话说清“不可删除的作用”。
- 前 3 章尤其强：主角、冲突、卖点要尽快落地。
- 至少每 2 到 3 章有明显升级或反转。
- 章末钩子不是生硬悬念，而是来自当前冲突。

返工信号：

- “继续修炼”“了解世界”“日常过渡”占太多。
- 多章重复同一功能。
- 章节之间缺少因果。
- 主角只旁观，不行动。
- 每章都靠新人物/新设定硬推。

## 8. 正文审核

正文生成后，不要先看辞藻，先看“可读性和推进”。

### 每章四问

```text
主角是否像主角？
剧情是否有推进？
信息是否前后一致？
章末是否让人想看下一章？
```

### 详细检查表

| 项目 | 审核问题 |
| --- | --- |
| 主角行动 | 主角是否主动观察、判断、选择？ |
| 人设一致 | 是否符合主角欲望、恐惧、缺陷？ |
| 金手指 | 是否仍有边界和代价？ |
| 冲突 | 是否有现实阻力，而非一路顺滑？ |
| 信息增量 | 这一章是否给了新信息？ |
| 情绪曲线 | 是否有期待、紧张、释放或反转？ |
| 文风 | 是否符合你要的口吻？ |
| 名词一致 | 人名、地名、境界、物品是否一致？ |
| 章末钩子 | 是否自然、具体、有压力？ |

通过标准：

- 这一章删掉会影响后续。
- 主角至少做了一次有意义的判断或选择。
- 有可感知的局势变化。
- 没有明显设定错误。
- 章末有明确“下一步想看什么”。

返工信号：

- 大段解释世界观，没有事件。
- 人物对话都像一个人。
- 主角突然变性格。
- AI 助手变成全知全能。
- 章节只是把大纲扩写成流水账。
- 章末靠“突然出现一个黑影”这种低成本悬念。

## 9. 章后管线审核

每章生成后，系统会做章后处理：

```text
摘要
关键事件
知识三元组
伏笔注册/消费
张力评分
文风漂移
人物状态
向量索引
```

你需要重点看三个地方。

### 9.1 章节摘要

检查：

- 摘要是否准确覆盖本章关键事件。
- 是否遗漏主角的重要选择。
- 是否把推测写成事实。
- 是否保留了后续要用的信息。

### 9.2 人物状态

检查：

- 人物关系是否变化。
- 角色是否知道不该知道的信息。
- 角色位置、伤势、境界、立场是否正确。
- 临时情绪不要误写成长期性格。

### 9.3 张力和文风

检查：

- 张力评分是否符合阅读感受。
- 文风漂移是否明显。
- 如果系统提示漂移，不要急着重写整章，先看是否是剧情需要。

## 10. 伏笔审核

伏笔要作为账本管理，不要让系统无限自动堆悬念。

每条伏笔至少要有：

```text
伏笔内容
埋设章节
预计回收章节/阶段
回收方式
当前状态
重要程度
```

分类建议：

| 类型 | 回收周期 | 例子 |
| --- | --- | --- |
| 短伏笔 | 1 到 3 章 | 一句话、一个异常动作、一个小道具。 |
| 中伏笔 | 5 到 20 章 | 身份线索、功法缺陷、关系秘密。 |
| 长伏笔 | 一卷或更久 | 世界真相、主角身世、大反派布局。 |

通过标准：

- 重要伏笔都有回收计划。
- 每隔几章有小回收，读者有反馈。
- 长伏笔不靠遗忘维持。

返工信号：

- 伏笔只埋不收。
- 太多神秘物品、神秘老人、神秘黑影。
- 回收和埋设没有因果。
- 回收时像临时编的。

## 11. 何时使用全自动模式

不建议一开始就开全自动。推荐策略：

```text
前 3 章：手动审
第 4 到 10 章：半自动，每章或每两章审一次
风格稳定后：可开全自动跑一小段
每 5 到 10 章：停下来总检
```

适合开启全自动的条件：

- Bible 已修过。
- 宏观规划已确认。
- 第一幕章节规划已确认。
- 前 3 章主角人设稳定。
- 知识库没有明显污染。
- 模型输出格式稳定。

不适合开启全自动的情况：

- 刚创建小说。
- 主角名/设定已经漂移。
- 模型经常输出格式错误。
- 大纲还很空。
- 你还没有确定文风。

## 12. 阶段性总检

建议节奏：

```text
第 1 章后：检查开局承诺
第 3 章后：检查主角、人设、类型卖点
第 5 章后：检查节奏和冲突升级
第 10 章后：检查第一幕闭环
每 5 到 10 章：检查知识库、伏笔、人物状态
每卷结束：检查主线、成长、未回收伏笔
```

### 10 章总检表

| 项目 | 审核问题 |
| --- | --- |
| 主线 | 这 10 章是否围绕一个明确目标？ |
| 主角 | 主角是否有成长或代价？ |
| 爽点 | 类型承诺是否兑现？ |
| 冲突 | 压力是否逐步升级？ |
| 配角 | 至少一个配角是否有记忆点？ |
| 反派/阻力 | 是否已经形成持续压力？ |
| 伏笔 | 是否有埋有收？ |
| 知识库 | 是否有错误事实？ |
| 下一阶段 | 读者是否知道接下来期待什么？ |

## 13. 常见问题判断

### 看起来卡住

先判断系统状态：

```bash
curl http://127.0.0.1:8005/health
curl http://127.0.0.1:8005/api/v1/autopilot/<novel_id>/status
```

关键字段：

| 字段 | 含义 |
| --- | --- |
| `daemon_process.running=false` | 守护进程没跑，自动驾驶不会推进。 |
| `current_stage=paused_for_review` | 正常等待人工确认。 |
| `needs_review=true` | 需要点“确认大纲，继续写作”。 |
| `autopilot_status=error` | 连续失败挂起，需要处理错误。 |
| 长时间有 LLM 请求日志 | 可能是在等模型返回，不一定卡死。 |

### 待审阅确认在哪里

进入小说工作台，找到自动驾驶面板，点击：

```text
确认大纲，继续写作
```

命令行也可以：

```bash
curl -X POST http://127.0.0.1:8005/api/v1/autopilot/<novel_id>/resume
```

### 章节开始变水

优先检查：

- 幕级章节规划是否太空。
- Context 里是否缺关键冲突。
- 知识库是否污染。
- 主角目标是否模糊。
- 每章是否缺“选择和代价”。

不要只靠“重写本章”，要回到章节规划或 Bible 修。

### 人名/设定漂移

处理顺序：

```text
修 Bible
修知识库三元组
修章节摘要
必要时修已生成正文
再继续自动驾驶
```

只改正文不改知识库，后面还会复发。

### 模型输出慢

可能原因：

- 规划输出很长。
- 模型网关慢。
- 超时设置偏小。
- Responses API 不支持后回退 Chat Completions。

日志中看到类似：

```text
Responses API unsupported ... falling back to chat completions
```

这通常不是致命错误，只是协议回退。

## 14. 推荐人工节奏

如果你想要“能看的小说”，建议这样分工：

| 阶段 | 人工参与度 |
| --- | --- |
| premise | 高，必须亲自写。 |
| Bible | 高，必须审。 |
| 宏观规划 | 高，必须审。 |
| 第一幕章节规划 | 高，必须审。 |
| 前 3 章正文 | 高，逐章审。 |
| 第 4 到 10 章 | 中，每 1 到 2 章审。 |
| 稳定后 | 中低，可开全自动跑短段。 |
| 每 5 到 10 章 | 高，做总检。 |
| 每卷末 | 高，检查主线和伏笔。 |

## 15. 快速通过/返工标准

### 可以继续自动驾驶

- 主角目标稳定。
- 章节规划有明确功能。
- 知识库没有明显错误。
- 伏笔数量可控。
- 文风基本稳定。
- 最近章节有推进和钩子。

### 应该暂停返工

- 主角人设变了。
- 名字、境界、地点连续出错。
- 大纲偏离最初卖点。
- 连续两章没有实质推进。
- 伏笔越来越多但没有回收。
- 金手指越来越万能。
- 模型开始输出思考过程或格式异常。

## 16. 一句话工作法

每次系统暂停时，不要问“能不能继续”，而是问：

```text
如果我现在让它继续写 5 章，当前设定、结构和记忆会不会把它带向正确方向？
```

答案是“会”，就确认继续。

答案是“不确定”，先修 Bible、结构、知识库或章节大纲。

答案是“不会”，不要继续生成，先返工。

## 01 项目介绍

### 项目介绍话术：
1. **项目背景**：1-2句话，项目的价值在于什么
2. **项目的架构**：
   - 关键组件（向量库、LLM、工具调用等）
3. **我负责的部分**：
   - 每个项目都要准备负责的部分描述，例如：方案设计、向量库的构建、模型的选型和优化
4. **项目的成果**：
   - 量化指标优先（如：准确率从 XX% 提升到 XX%，响应时间降低 XX%）
   - 没有量化指标则描述业务价值

### 数据流转话术
从用户的输入开始，举例子，每经过一个节点/模型/处理都要详细说明，最终输出的东西也要举例子。

**示例模板**：
> 用户输入"帮我查一下XX合同的付款条款" → 经过意图识别模块判断为文档检索类问题 → Query 改写为更规范的检索语句 → 向量化后在 Milvus 中 Top-K 检索 → 召回 3 段相关文本 → 拼接 Prompt 送入 Qwen-7B → 输出"该合同第5条规定付款周期为30天"

### 必备的基础知识（严禁回答出错）：

1. **项目中所有模型的型号，回答要专业（如：Qwen2.5-7B-Instruct、BGE-M3、Reranker-v2 等）**
2. **有哪些算力资源，训练时间（如：使用 2×A100 80G，微调耗时约 4 小时）**
3. **项目如何做评估，测试集是怎么来的。**
     1. **人工评估**
     2. **大模型评估**
     3. **人工+大模型评估**
4. **每个数据库里存储的数据，请举例说明，每个数据库存储的数据量?**
5. **如果有意图识别，意图识别怎么实现**
6. **每个调用大模型地方都要写一个提示词的案例**

## 03 模型选型及并发参考

### BERT 模型微调的设计

**严禁设计 BERT 模型用 LoRA 微调**，技术上没问题，但实际工程中不这么做。

原因：BERT 参数量小（110M~340M），全量微调成本极低，没有必要用 LoRA 节省参数；LoRA 主要用于解决大模型（7B+）全量微调显存不足的问题。

### 如何选型：

**按任务类型选型：**

| 任务类型            | 推荐模型(仅作为例子)                    | 理由                       |
| --------------- | ------------------------------ | ------------------------ |
| 文本分类/NER/意图识别   | BERT / RoBERTa / MacBERT       | Encoder-only，推理快，适合判别式任务 |
| 文本生成/对话/摘要      | Qwen2.5 / GLM-4 / LLaMA3       | Decoder-only，生成能力强       |
| 向量检索（Embedding） | BGE-M3 / text-embedding-3      | 专为语义检索优化                 |
| 重排序（Rerank）     | BGE-Reranker-v2 / bce-reranker | 精排阶段提升召回质量               |
| 多模态任务           | Qwen-VL / InternVL             | 支持图文理解                   |

### 大参数模型生成式任务私有化部署选型

> 核心约束：需要私有化部署（数据不出内网）+ 生成式任务 + 效果优先

**推荐模型（按效果排序）：**

| 模型                    | 参数量       | 最低显存需求                               | 推荐部署方式        | 特点                   |

## 04 项目中的记忆模块是怎么设计

记忆模块是对话系统的核心组件，面试时要能说清楚**记忆的分类**和**工程实现**。

### 记忆的分类

| 类型            | 说明        | 实现方式                                |
| ------------- | --------- | ----------------------------------- |
| 短期记忆（对话历史）    | 当前会话内的上下文 | （Memory）直接拼接到 Prompt 的 messages 列表中 |
| 长期记忆（用户画像/知识） | 跨会话持久化信息  | 存入向量数据库或关系型数据库，按需检索                 |
| 工作记忆（任务状态）    | 当前任务的中间状态 | 存入 Agent 的 state 字典或 Redis          |

### 短期记忆设计要点

```
问题：上下文窗口有限，历史消息不能无限堆叠
解决方案：
  1. 滑动窗口：只保留最近 N 轮对话
  2. 摘要压缩：超出窗口的历史用 LLM 压缩成摘要，摘要替代原始消息
  3. Token 预算控制：按 Token 数截断，而非轮数
```

### 长期记忆设计要点

```
存储：用户偏好、历史结论、重要事件 → 向量化后存入 Milvus/Chroma
检索：每次对话开始时，用当前 Query 检索相关长期记忆，注入 System Prompt
更新：对话结束后，提取关键信息写回长期记忆（可用 LLM 做信息抽取）
```

### 面试回答模板

> 我们项目中的记忆分两层：短期记忆直接维护 messages 列表，超过 2000 Token 时触发摘要压缩；长期记忆存在 Milvus 里，每轮对话开始时检索 Top-3 相关记忆注入 System Prompt，对话结束后用 LLM 抽取关键信息异步写回。

## 05 你对 RAG 了解比较多，请你详细说一下 RAG 这个系统

### RAG 的完整流程

RAG（Retrieval-Augmented Generation，检索增强生成）= 检索系统 + 生成模型。

```
离线阶段（构建知识库）：
原始文档 → 文档解析（PDF/Word/HTML）→ 文本分块（Chunking）
→ Embedding 向量化 → 存入向量数据库（Milvus/Chroma/ES）

在线阶段（用户查询）：
用户 Query → [可选] Query 改写/扩展 → Embedding 向量化
→ 向量检索 Top-K → [可选] Rerank 精排 → 拼接 Prompt
→ LLM 生成回答 → 输出（含引用来源）
```

### 关键模块详解

**1. 文档分块（Chunking）策略**
- 固定长度分块：简单，但可能切断语义
- 语义分块：按段落/句子边界切，保留语义完整性
- 滑动窗口：chunk 之间有重叠（overlap），避免关键信息被切断
- 推荐**（按照实际的来）**：chunk_size=512 token，overlap=50 token

**2. 向量检索**

- 稠密检索（Dense）：BGE/text-embedding 模型，语义相似度高
- 稀疏检索（Sparse）：BM25，关键词匹配，适合专有名词
- 混合检索（Hybrid）：稠密 + 稀疏融合，效果最佳（RRF 融合算法）

**3. Rerank 重排序**
- 召回阶段 Top-K 较大（如 Top-20），Rerank 后取 Top-3~5
- 使用 Cross-Encoder 模型（如 BGE-Reranker），对 Query-Doc 对打分
- 显著提升最终召回质量，是 RAG 效果提升的关键步骤

**4. Prompt 构建**

```
System: 你是一个专业助手，请根据以下参考资料回答问题，如果资料中没有相关信息，请如实说明。
参考资料：
[1] {chunk_1}
[2] {chunk_2}
[3] {chunk_3}
User: {用户问题}
```

### RAG 常见问题及优化方向

| 问题          | 原因             | 优化方案                  |
| ----------- | -------------- | --------------------- |
| 召回不到相关内容    | Query 与文档表述差异大 | Query 改写、HyDE（假设文档嵌入） |
| 召回内容不精准     | 向量检索粒度粗        | 加 Rerank、混合检索         |
| LLM 没用上召回内容 | Prompt 设计问题    | 优化 Prompt，强调使用参考资料    |
| 回答有幻觉       | LLM 自身问题       | 加引用溯源，要求 LLM 标注来源     |

## 07 给你一个多 Agent 的复杂任务，请问你怎么对这个任务做拆解

### 任务拆解的核心思路

面试时要展示**系统性思维**，不要直接说"分成几个 Agent 去做"，要说清楚**为什么这么拆**。

### 拆解步骤

**第一步：理解任务，识别子目标**

- 把复杂任务分解为若干个独立的、可验证的子目标
- 判断子目标之间的依赖关系（串行 vs 并行）

**第二步：设计 Agent 角色**

```
常见 Agent 角色：
- Planner Agent（规划者）：负责任务分解和调度，是整个系统的大脑
- Executor Agent（执行者）：负责具体子任务执行（搜索、计算、写代码等）
- Critic Agent（评审者）：负责检查执行结果是否符合预期
- Memory Agent（记忆者）：负责维护任务状态和上下文
```

**第三步：设计任务流**

```
示例：用户要求"帮我调研竞品并生成分析报告"

Planner 拆解：
├── Task1: 搜索竞品列表（Executor-Search）
├── Task2: 并行抓取各竞品官网信息（Executor-Crawl × N）
├── Task3: 对每个竞品做结构化分析（Executor-Analyze）
├── Task4: Critic 检查分析是否完整
└── Task5: 汇总生成报告（Executor-Write）

依赖关系：Task1 → Task2 → Task3 → Task4 → Task5
Task2 内部各竞品可并行
```

**第四步：异常处理设计**
- 子任务失败时的重试机制
- 超时处理
- 人工介入节点（Human-in-the-loop）

### 面试回答模板

> 拿到复杂任务后，我会先用 Planner Agent 做任务分解，识别出哪些子任务可以并行、哪些必须串行。设计输出方案文档，然后为每类子任务分配专门的 Executor Agent，并设置 Critic Agent 做结果校验。整个流程用langgraph来表示任务依赖。

## 08 Function Call、MCP 和 Skill 的区别

这三个概念都是让 LLM 具备"调用外部能力"的机制，但层次不同。

### Function Call（函数调用）

**定义**：OpenAI/各大模型厂商提供的原生能力，让 LLM 在生成回复时，能够输出结构化的"我要调用某个函数"的指令。

**工作原理**：
```
1. 开发者在请求中定义 tools（函数描述 + 参数 schema）
2. LLM 判断是否需要调用工具，如需要则输出 tool_call（函数名 + 参数）
3. 开发者代码执行该函数，将结果返回给 LLM
4. LLM 根据函数结果生成最终回复
```

**特点**：
- 最底层的机制，由模型原生支持
- 开发者需要自己实现函数逻辑和调用循环
- 适合：精确控制工具调用逻辑的场景

### MCP（Model Context Protocol）

**定义**：Anthropic 提出的开放协议，标准化了 LLM 与外部工具/数据源之间的通信方式。

**工作原理**：

```
MCP Server：提供工具、资源、Prompt 模板（独立进程/服务）
MCP Client：LLM 应用，通过标准协议调用 MCP Server
通信方式：stdio 或 HTTP/SSE
```

**特点**：
- 是一个**协议标准**，不是具体实现
- 解决了工具生态碎片化问题：一个 MCP Server 可被多个 AI 应用复用
- 类比：MCP 之于 AI 工具，就像 USB 之于硬件外设
- 适合：构建可复用的工具生态，多个 Agent/应用共享同一套工具

### Skill（技能）

**定义**：在 Agent 框架层面的概念，是对一组能力的封装，通常包含多个 Function Call 的组合逻辑。

**特点**：
- 比 Function Call 更高层的抽象
- 一个 Skill 可能内部调用多个工具，包含完整的业务逻辑
- 可复用、可组合，是 Agent 能力的基本单元
- 适合：构建复杂 Agent 系统时，对能力进行模块化管理

### 三者对比总结

| 维度 | Function Call | MCP | Skill |
|-----|--------------|-----|-------|
| 层次 | 模型原生能力 | 通信协议标准 | 框架层能力封装 |
| 粒度 | 单个函数 | 工具服务集合 | 业务能力模块 |
| 复用性 | 低（需重复定义） | 高（跨应用复用） | 中（框架内复用） |
| 标准化 | 各厂商不统一 | 开放标准协议 | 框架自定义 |
| 典型场景 | 调用天气 API | 提供文件系统/数据库访问服务 | "搜索并总结"复合能力 |

### 面试回答模板

> Function Call 是模型层的原生能力，让 LLM 能输出结构化的工具调用指令；MCP 是 Anthropic 提出的协议标准，解决了工具生态碎片化的问题，让一套工具服务可以被多个 AI 应用复用，类似 USB 协议；Skill 是 Agent 框架层的概念，是对多个工具调用和业务逻辑的高层封装，是 Agent 能力的基本单元。三者是不同层次的概念，实际项目中往往同时用到：用 MCP 暴露工具服务，用 Function Call 让模型调用，用 Skill 封装复合业务逻辑。









## 09 当Agent需要频繁查询底层业务关系型数据库来回答用户问题(Text-to-SQL)时，如何设计数据库的连接池、索引策略以及缓存机制，以大幅降低查询延迟并防止数据库挂掉?

### 连接池设计

- 使用连接池（SQLAlchemy Pool / pgBouncer / HikariCP），避免每次查询都建立新连接
- 关键参数：`pool_size=10~20`，`max_overflow=5`，`pool_timeout=30s`，`pool_recycle=1800s`（防止连接超时失效）
- Agent 并发查询时从池中借连接，用完归还，不要每次 new connection

### 索引策略

- 对 `WHERE` 子句高频字段建单列索引（如 `order_date`、`user_id`）
- 对多条件查询建复合索引，遵循**最左前缀原则**（如 `(dept_id, status, created_at)`）
- 对 `JOIN` 的关联字段必须建索引
- 避免在低基数字段（如性别、状态枚举）上单独建索引，效果差
- 定期分析慢查询日志（`EXPLAIN ANALYZE`），针对性补索引

### 缓存机制（三级缓存）

```
L1 本地内存缓存（LRU）：
  - 缓存最近 100 条高频 SQL 的结果，命中直接返回，延迟 <1ms
  - 适合：同一进程内重复查询

L2 Redis 分布式缓存：
  - 缓存查询结果，Key = SQL 语句的 MD5，TTL 按业务设置（如实时数据 30s，历史数据 10min）
  - 适合：多实例部署时共享缓存

L3 语义缓存（Semantic Cache）：
  - 将历史问题向量化存入向量库，新问题与历史问题做相似度匹配
  - 相似度 > 0.95 时直接复用历史 SQL 和结果，无需重新生成 SQL
  - 适合：用户问法不同但语义相同的重复查询
```

### 防止数据库挂掉

- **只读副本**：Agent 查询全部走只读副本（Read Replica），不影响主库写入
- **限流**：对 Agent 的查询 QPS 做限制（如令牌桶算法，最多 50 QPS）
- **超时控制**：每条 SQL 设置执行超时（如 5s），超时直接返回错误，不阻塞连接
- **熔断机制**：查询失败率超过阈值（如 50%）时触发熔断，返回降级提示，保护数据库
- **SQL 安全校验**：Text-to-SQL 生成的 SQL 必须做白名单校验，只允许 SELECT，禁止 DDL/DML

### 面试回答模板

> **Redis 缓存通用结果**，语义缓存复用相似问题的历史 SQL。稳定性方面，查询全走只读副本，**加 QPS 限流**和 SQL 执行超时，同时对生成的 SQL 做安全校验，只允许 SELECT 语句执行。



## 10 RAG的文本分块如何保证文本语义完整性

### 核心策略

**1. 按自然边界切分（优先）**

- 优先按段落 → 句子边界切，而非固定字符数截断
- 保留标题、小节等结构信息，每个 chunk 携带所属章节的元数据

**2. 滑动窗口（Overlap）**
- 相邻 chunk 之间保留重叠部分（推荐 overlap = chunk_size 的 10%~20%）
- 例：chunk_size=512 token，overlap=50 token
- 作用：避免关键信息恰好落在两个 chunk 的边界被切断

**3. 语义分块（Semantic Chunking）**

- 用 Embedding 模型计算相邻句子的语义相似度
- 相似度突变处（余弦相似度骤降）作为切分点
- 效果最好，但计算成本较高，适合离线构建知识库

**4. 层级分块（Hierarchical Chunking）**

- 大 chunk（512~1024 token）存摘要/主题，小 chunk（128~256 token）存细节
- 检索时先用大 chunk 定位主题，再用小 chunk 精确召回
- 适合长文档（报告、合同、手册）

**5. 特殊内容不拆分**

- 表格、代码块、列表等结构化内容整体作为一个 chunk，不在内部切分
- 跨页的同一段落合并后再切分

### 面试回答模板

> 保证语义完整性主要靠三点：第一，优先按段落和句子边界切，不做硬截断；第二，设置 overlap（如 50 token 的重叠），防止关键信息落在 chunk 边界；第三，对于效果要求高的场景，用语义分块——计算相邻句子的 embedding 相似度，在语义突变处切分。表格、代码块这类结构化内容不拆分，整体作为一个 chunk 处理。

## 11 Embedding 模型如何训练

### 训练目标

让语义相似的文本在向量空间中距离近，语义不同的文本距离远。核心是**对比学习（Contrastive Learning）**。

### 训练流程

**第一步：构建训练数据（正负样本对）**

| 样本类型 | 来源示例 |
|---------|---------|
| 正样本（语义相似对） | 问答对、同义句、段落-摘要对、标题-正文对 |
| 难负样本（Hard Negative） | 语义相近但实际不相关的句子（如同领域不同主题） |

> 难负样本是提升 Embedding 质量的关键，普通随机负样本太容易区分，模型学不到细粒度语义。

**第二步：选择基座模型**
- 通常基于qwen3-0.6b-embbding  4b   8b bge   金融领域emdedding  

**第三步：对比学习微调**

```
损失函数：InfoNCE Loss（对比损失）
  - 对于一个 Query，正样本相似度要高，所有负样本相似度要低
  - 温度系数 τ 控制分布的尖锐程度（通常 0.05~0.1）

训练方式：
  - In-batch Negative：同一 batch 内其他样本作为负样本，简单高效
  - Hard Negative Mining：专门挖掘难负样本，效果更好
```



**第四步：评估**

- 通用评估：MTEB 基准测试（检索、分类、聚类等多任务）
- 业务评估：在实际检索任务上测 Recall@K、MRR

### 面试回答模板

> Embedding 模型的训练核心是对比学习。构建正样本对（如问答对、同义句）和难负样本，用 InfoNCE Loss /对比损失训练，让相似文本的向量距离近、不相似的距离远。难负样本的质量直接决定模型效果。

## 12 LangChain 和 LangGraph 有什么区别

### LangChain

- **定位**：链式调用框架，适合**线性流程**
- **核心抽象**：Chain（链）、Agent、Tool、Memory
- **执行模式**：顺序执行，A → B → C，流程固定
- **适合场景**：简单 RAG、单轮工具调用、线性 Pipeline

### LangGraph

- **定位**：基于**有向图（DAG/循环图）**的 Agent 编排框架，是 LangChain 的扩展
- **核心抽象**：Node（节点）、Edge（边）、State（全局状态）
- **执行模式**：支持**循环、条件分支、并行执行**
- **适合场景**：复杂多 Agent 系统、需要反复推理的任务、有状态的长流程

### 核心区别对比

| 维度 | LangChain | LangGraph |
|-----|-----------|-----------|
| 流程结构 | 线性链式 | 图结构（支持循环） |
| 状态管理 | 隐式，各组件自管理 | 显式 State 对象，全局共享 |
| 循环支持 | 不支持（Agent 内部有循环但不透明） | 原生支持，可定义循环终止条件 |
| 并行执行 | 有限支持 | 原生支持并行节点 |
| 调试可视化 | 较弱 | 图结构天然可视化，易调试 |
| 适用复杂度 | 简单到中等 | 中等到复杂 |

### 面试回答模板

> LangChain 是链式框架，适合线性流程，比如 RAG 的"检索→拼接→生成"这种固定流程。LangGraph 是图结构的 Agent 编排框架，核心优势是支持循环和显式状态管理——Agent 可以反复调用工具直到满足终止条件，整个流程的状态用一个 State 对象统一管理，更适合复杂的多 Agent 系统。实际项目中，简单流程用 LangChain 够了，复杂的 Agent 编排推荐用 LangGraph。

## 13 BM25 和 TF-IDF 的区别

### TF-IDF

- **TF**（词频）：词在文档中出现的次数 / 文档总词数
- **IDF**（逆文档频率）：log(总文档数 / 含该词的文档数)，衡量词的稀有程度
- **得分** = TF × IDF
- **问题**：TF 没有上限，词频越高分数线性增长，不合理（一篇文章里"苹果"出现 100 次，不代表比出现 10 次的文章相关 10 倍）

### BM25（Best Match 25）

BM25 是 TF-IDF 的改进版，是信息检索领域的经典算法，也是 **Elasticsearch 的默认相关性算法**。

**两个核心改进：**

1. **TF 饱和（Saturation）**：用非线性函数限制 TF 的增长，词频增加到一定程度后收益递减
   - 参数 `k1`（通常 1.2~2.0）：控制 TF 饱和速度

2. **文档长度归一化**：长文档中词频自然更高，BM25 对此做惩罚
   - 参数 `b`（通常 0.75）：控制长度归一化程度，b=0 不归一化，b=1 完全归一化

**公式对比：**
```
TF-IDF：score = TF × IDF

BM25：score = IDF × [TF × (k1 + 1)] / [TF + k1 × (1 - b + b × docLen/avgDocLen)]
```

### 对比总结

| 维度 | TF-IDF | BM25 |
|-----|--------|------|
| TF 上限 | 无上限，线性增长 | 有饱和，增长递减 |
| 文档长度 | 不考虑 | 做归一化处理 |
| 实际效果 | 较弱 | 显著优于 TF-IDF |
| 应用场景 | 基础文本相似度 | 搜索引擎、RAG 稀疏检索 |

### 面试回答模板

> BM25 是 TF-IDF 的改进版。TF-IDF 的问题是词频越高，分数线性增长越多，不合理；BM25 用非线性函数做 TF 饱和处理，词频增加到一定程度后收益递减，同时加入文档长度归一化，避免长文档因词频自然偏高而占优势。

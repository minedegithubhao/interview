# **dify简历**

## 项⽬背景：

基于dify构建的一款面向求职全流程的智能辅助系统，旨在帮助用户从简历准备、岗位匹配、企业风险识别、面试训练到 Offer 决策，完成更高效、更精准、更安全的求职过程。

## 技术架构

本项目采用 **Chatflow + Workflow + RAGFlow** 的组合式智能体架构。整体以 **Chatflow** 作为对话交互与任务调度骨架，负责理解用户意图、拆解求职场景任务，并根据不同需求调用对应能力模块；各功能模块通过多个 **Workflow** 封装为可复用工具链，分别支撑8大核心能力；同时接入 **RAGFlow 外部知识库**，用于管理行业薪资数据、企业信息、面试题库、话术模板，实现基于知识增强的智能问答、内容生成与决策分析。

整体架构具备良好的模块化、可扩展性和知识可更新能力，能够支持后续新增更多求职场景工具和行业知识库。

## 功能介绍：

本项目包括八大模块：

1. 简历诊断与优化
2. 根据岗位需求一键定制简历
3. 企业风险识别
4. offer评估与薪资收益计算
5. 面试语音复盘
6. 面试模拟训练
7. 求职话术
8. 行业薪资报告。

## 主要流程

1. 用户输入
2. 用户输入整理器：整理用户真实需求，不扩展、不推理、不回答，为后续意图识别提供稳定、准确的输入。同时完成规范化表达、异常过滤、隐私保护、安全校验和权限边界控制，为后续Agent 调度提供可靠输入
3. 意图识别：根据用户输入整理器的输出，识别用户当前的核心意图，同时对无法处理的需求进行兜底，避免触发不合规操作。
4. 处理节点：根据意图识别结果，路由至对应的业务处理流程，并调用已封装的 Workflow 提供具体服务。包括但不限于工具调用、第三方插件、文档信息提取、条件判断、LLM 推理、知识库检索、代码块执行等确保不同求职场景能够被准确、高效、可控地处理。
5. 记忆与回复：使用会话变量保存用户的上下文信息，从而避免了llm丢失上下文

## 主要职责

- **Dify 与 RAGFlow 工作空间搭建**：由于团队成员本地开发环境不统一，项目中容易出现工作流版本混乱、知识库无法复用、配置不一致、无法协同调试等问题，影响整体协作效率和交付进度。为解决这一问题，我负责搭建 **Dify 与 RAGFlow** 的共享工作空间，统一工作流编排、知识库管理和工具调用方式，使团队成员可以直接通过 Web 端进行开发、调试和验证。最终，该方案减少了团队成员各自搭建和维护本地环境的开发成本，降低了因环境差异导致的问题排查成本，同时提升了多人协作效率、开发规范性和项目交付一致性。
- **工作流工具化封装与版本管理**：由于这些 Workflow 会被多个 Agent 和业务场景重复调用，如果每个场景单独配置，容易出现重复建设、调用方式不统一和版本混乱的问题。为了解决这个问题，我统一了 Workflow 的输入输出和调用方式，并将其封装成可复用工具，同时引入版本管理机制，对工具版本进行明确标识和维护。最终减少了重复配置成本，也避免了团队不知道当前使用的是新版本还是旧版本的问题，提升了协作效率和结果一致性。
- **长语音转文本工具建设：**由于 Dify 官方模型市场未原生接入阿里 ASR，若通过 HTTP 节点接入，不仅门槛较高，还需要公网可访问 URL 和 OSS 等对象存储中转；同时 Dify 原生上传能力对文件大小和时长有限制，较长语音容易失败。为解决这一问题，我基于 `faster-whisper` 搭建了本地化长语音转写能力，采用文件切分、多线程并行处理、Overlap 上下文衔接和结果 Merge 合并的方案，提升转写效率并保证分段边界处的语义连续性。最终工具支持最长 **1 小时、100MB** 的语音文件转写，可支撑面试录音转写、面试复盘和语音材料整理等业务场景。
- **提供稳定的rag检索服务**：由于面试题库是典型的 FAQ 型短问题，存在“问题短、答案长”的特点，用户提问时又经常使用简称、别名和口语化表达，导致用户语言和题库标准问题不一致，长答案也容易稀释核心问题语义。为了解决这个问题，我从查询侧和文档侧同时优化：做查询归一化、术语对齐、同义词扩展和关键词增强提升了 RAG 检索的召回率和稳定性，通过多路召回、RRF 融合和 reranker 精排提升准确率。最终为面试题库问答、模拟面试和面试复盘提供了更稳定的检索支撑。

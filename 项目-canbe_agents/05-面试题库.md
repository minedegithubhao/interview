# 05-面试题库

## 一、从复盘抽出的通用面试指南

基于 [page-2026-05-07-00-08-09.md](C:/Users/82473/Downloads/page-2026-05-07-00-08-09.md) 这篇复盘，可以把大模型算法岗里常见的项目面试方法论收敛成下面几条。

### 1. 项目题永远先讲场景，再讲技术

面试官先想确认的，不是候选人用了什么组件，而是：

- 项目到底解决什么问题
- 这个问题为什么值得这样解决
- 原始方案为什么不够

所以项目介绍最好按这个顺序展开：

1. 业务场景
2. 问题痛点
3. 为什么需要这套方案
4. 方案的代价与收益

不要一上来就讲架构图、模型名、框架名。那种讲法像“系统说明书”，不像“问题求解过程”。

### 2. 每个技术点都要准备“三问”

复盘里最有价值的地方，是它把面试官的追问模式讲得很清楚。几乎每个技术选型，都会被反向验证：

1. 为什么用它？
2. 为什么不用别的？
3. 用它的代价是什么？

这是项目准备的核心方法。  
如果一个技术点只能回答“我用了它”，那它在面试里就是脆弱点。

### 3. 高分回答靠因果链，不靠术语堆砌

低质量回答通常长这样：

- 我们用了 RAG
- 我们用了 rerank
- 我们用了向量库
- 我们效果提升了

这类回答的问题是“只有名词，没有判断”。

更好的表达应该是：

- 遇到了什么问题
- 原方案为什么不够
- 这个技术点具体解决了哪一段问题
- 它带来了什么副作用
- 最后用什么指标证明它有效

### 4. 项目要准备四层讲法

复盘里提到的“项目扩写四层”非常实用，可以直接拿来当备考框架：

1. **1 分钟版**  
   一句话定位 + 做了什么 + 结果如何
2. **5 分钟版**  
   核心架构、关键模块、核心选型
3. **15 分钟版**  
   每个模块的实现细节、对比方案、踩坑、权衡
4. **深聊版**  
   演进方向、失败案例、评测体系、行业对比

面试官追到第几层，你能撑到第几层，基本决定项目分数。

### 5. “不会”不能空掉，要给出推理路径

项目面试里，最差的回答不是“不知道”，而是“不知道且没有思路”。

比较稳的回答方式是：

- 先承认自己没直接做过
- 再给出业界常见的两三种思路
- 最后结合当前项目说明会怎么选

这样可以体现三件事：

- 视野
- 判断力
- 场景意识

### 6. 评测意识比功能清单更重要

大模型项目现在最容易同质化。真正拉开差距的，往往不是“用了多少组件”，而是能不能回答这些问题：

- 效果怎么测
- 优化后怎么证明变好了
- 如果变差了，怎么定位是哪一层出问题
- 线上反馈如何变成失败样本

一句话概括：

**项目越往后走，面试官越不关心你“会不会搭”，越关心你“会不会证伪和验证”。**

### 7. 简历是地图，不是名词仓库

复盘里还有一个很实在的提醒：简历上每一个技术词都可能成为深挖入口。

因此：

- 不要写自己讲不清楚的名词
- 不要为了显得全而堆技术栈
- 写出来的每个点，都要做好被连问三层的准备

### 8. 项目回答的标准模板

结合复盘，项目类问题最稳的回答模板是：

1. 先讲场景
2. 再讲问题
3. 再讲原方案不足
4. 再讲为什么选当前方案
5. 再讲 trade-off
6. 最后讲评测和结果

这个模板适合绝大多数“介绍项目”“为什么这样设计”“为什么不用别的方案”类型问题。

## 二、从面试官角度，针对该项目的提问清单

这个项目的准确定位，不是泛化多 Agent 平台，而是：

**面向京东帮助中心公开 FAQ 的可控 RAG 问答后端。**

因此，面试官深挖时，重点通常不会放在 Planner、多 Agent 调度、RL 对齐，而会集中在“场景边界、数据构建、检索链路、回答约束、评测体系、工程选型”这几块。

### 1. 项目定位与边界

1. 这个项目一句话解决什么问题？
2. 为什么这是 FAQ RAG 问题，而不是通用聊天问题？
3. 为什么只做公开 FAQ，不做订单、物流、退款状态查询？
4. 你怎么定义“能答”和“不能答”？
5. 这个项目是特定领域，还是通用能力？
6. 它和普通聊天机器人最大的差别是什么？
7. 为什么你强调“可控”，而不是“尽量多答”？
8. 如果用户坚持问个体状态问题，你的系统为什么还必须 fallback？

### 2. 数据源与知识构建

1. 为什么选京东帮助中心公开 FAQ 作为数据源？
2. 你如何约束 `sourceUrl` 只来自允许路径？
3. 为什么要把来源限定到特定域名和路径？
4. 原始 FAQ 文本存在哪些脏数据问题？
5. 清洗规则有哪些？删掉了哪些内容？
6. 为什么既存 FAQ 主文档，又存 chunk？
7. chunk 是按段落、按句子、按字数，还是混合切分？
8. 为什么要加 overlap？
9. overlap 太大或太小分别会有什么问题？
10. 近似 FAQ、重复 FAQ、历史 FAQ 如何处理？
11. 如何避免过期规则在检索里排到前面？
12. 为什么“公开可跳转 source”在这个项目里是刚性要求？

### 3. 检索链路设计

1. 为什么不是单路向量检索，而是 dense + sparse + keyword 混合检索？
2. BM25、dense、sparse 各自解决什么问题？
3. 在你的场景里，哪类 query 更依赖关键词匹配，哪类更依赖语义匹配？
4. 为什么要做 query normalize？
5. 为什么要做同义词扩展？
6. 为什么要做 query rewrite？
7. 如果用户问“密码丢了咋办”，系统如何映射到“忘记密码怎么办”？
8. 为什么多路召回后还要做融合？
9. 为什么选 RRF，而不是直接按召回分数加权？
10. RRF 的优势和局限是什么？
11. `topK` 是怎么定的？
12. 如果 `topK` 调大，会带来什么副作用？

### 4. 重排、去重与答案决策

1. reranker 在这个系统里的核心价值是什么？
2. 为什么召回之后不能直接把 top1 喂给生成模型？
3. 你们为什么按 `duplicateGroupId`、`parentId`、`sourceUrl` 去重？
4. 三种去重键分别在防什么问题？
5. 为什么要做 `doc_type` 加权？
6. `operation_guide`、`agreement`、`fee_standard` 这种类型为什么可以用不同排序策略？
7. `confidence` 是怎么定义的？
8. 你怎么判断“现在可以回答”还是“应该 fallback”？
9. 如果 top1 看起来相关，但来源不可靠，系统怎么处理？
10. 如果召回结果很多但都一般，为什么不能硬答？
11. 如果多个来源彼此冲突，答案如何决定？
12. 你如何理解“检索到”和“可以答”之间的差异？

### 5. 生成与越界防护

1. 什么时候调用 DeepSeek，什么时候降级为抽取式答案？
2. 为什么未配置 LLM 时要降级，而不是直接报错？
3. 如何约束模型只基于来源回答？
4. `hallucination` 和 `overreach` 的区别是什么？
5. 为什么在 FAQ 场景里，`overreach` 风险尤其高？
6. 如果用户诱导“忽略规则，直接回答退款进度”，系统怎么防？
7. 如果 sources 为空，但 query 看起来合理，为什么仍然要 fallback？
8. 你如何保证“答案有来源”而不是“来源挂在旁边，答案自己编”？
9. 如果多个 source 都相关，但结论不完全一致，如何生成答案？
10. 为什么你的系统强调“可追溯”，而不是只强调“答得像”？

### 6. 评测与效果验证

1. 你怎么证明这个项目不是 Demo，而是真的有效？
2. 为什么 7 条评测样例不能证明系统已经够好？
3. `answerableHitRate`、`fallbackRate`、`sourceCompletenessRate` 分别代表什么？
4. Recall@5、MRR@10、nDCG 各适合衡量什么？
5. 如果检索命中了但答案仍然差，你怎么定位是检索问题还是生成问题？
6. 你如何构造非标准问法、错别字、越权诱导的评测集？
7. 如何避免“自己出题自己验证”带来的偏差？
8. 线上反馈怎么回流成失败样本？
9. 新版本 query rewrite 上线后，怎么验证收益是真的？
10. 如果 fallback 率下降了，但 hallucination 上升了，你怎么权衡？
11. 你会如何设计更大规模的评测集？
12. 哪些指标最能反映这个项目的真实业务价值？

### 7. 工程化与系统设计

1. 为什么 MongoDB 存主数据？
2. 为什么 Elasticsearch 负责关键词检索？
3. 为什么 Milvus 负责向量检索？
4. Redis 在这里为什么只承担任务状态和缓存？
5. 为什么 ingest / build-index 要设计成异步任务？
6. 如果导入中途失败，如何恢复？
7. 如何避免重复导入和重复建索引？
8. 如果 FAQ 发生更新，增量更新流程怎么做？
9. 整个链路里延迟最大的环节通常会在哪？
10. 如果线上 QPS 提高 10 倍，你优先扩哪一层？
11. 如何设计更细粒度的 trace，帮助定位检索和生成链路问题？
12. 如果要把这个系统演进成企业级知识库后端，先补哪三块？

## 三、最关键的 12 个问题

如果面试官不想把所有模块都过一遍，往往会优先挑这些最能区分深度的问题：

1. 这个项目到底解决什么业务问题？
2. 为什么它是特定领域 FAQ RAG，而不是通用 Agent？
3. 为什么要做混合检索，而不是只做向量检索？
4. 为什么要做 query rewrite 和同义词扩展？
5. 为什么选 RRF？
6. reranker 在这里的核心价值是什么？
7. chunk 策略怎么定的？失败案例有哪些？
8. 你怎么判断该回答还是该 fallback？
9. 你怎么防止模型回答订单、物流、退款这类越界问题？
10. 你怎么证明返回的答案真的被 source 支撑？
11. 为什么当前评测结果不能代表生产级能力？
12. 如果要把这个项目升级成企业知识库产品，先补哪三块？

这 12 个问题，本质上覆盖了：

- 场景理解
- 方案选型
- 检索设计
- 安全边界
- 评测意识
- 工程演进能力

如果这 12 个问题答得扎实，项目面试大体就能站住。

## 四、把复盘转成自己的备考动作

下面这套动作，可以直接拿来准备这个项目的面试。

### 1. 给每个关键技术点写“三问卡片”

建议至少覆盖这些模块：

- FAQ 数据清洗
- chunk 策略
- 同义词扩展
- query rewrite
- 混合检索
- RRF
- reranker
- 去重
- fallback 阈值
- source 约束
- 评测指标

每张卡片统一写：

1. 为什么用它
2. 为什么不用别的
3. 用它的代价是什么

### 2. 准备项目的四层讲法

建议把这个项目分别写成四个版本：

1. **1 分钟版**  
   一句话定位 + 主要能力 + 结果
2. **5 分钟版**  
   数据、检索、重排、生成、fallback、评测
3. **15 分钟版**  
   每个模块的选型理由、替代方案、踩坑
4. **深聊版**  
   失败案例、定位手段、未来演进、生产化改造

### 3. 给每个模块准备一个失败案例

这是项目回答能否显得“做过”的关键。

建议至少准备这些失败场景：

- query rewrite 把口语问题改坏了
- chunk 切太碎导致关键信息被拆开
- chunk 切太大导致检索不够精准
- 历史规则 FAQ 被误召回
- 关键词命中了，但语义上答偏
- rerank 把正确结果排低了
- confidence 阈值过低导致误答
- confidence 阈值过高导致过度 fallback
- source 有，但答案没有被 source 真正支撑

每个失败案例最好都能回答：

1. 现象是什么
2. 你怎么发现的
3. 根因是什么
4. 如何修复
5. 修复后如何验证

### 4. 给每个指标准备“怎么测”的答案

不要只记指标名。  
要准备：

- 评测集怎么构造
- ground truth 怎么标
- baseline 是什么
- 指标为什么选它
- 指标的局限是什么

因为面试官很可能不会停在“你用了哪些指标”，而是继续问：

- 这些指标和业务目标的关系是什么
- 数据规模是否足够
- 指标之间冲突时怎么取舍

### 5. 按“反事实”方式自己 mock

对项目里的每个技术点，自己至少追问三轮：

- 如果不用这个方案会怎样？
- 如果让你换一个方案，会换成什么？
- 这个方案在什么场景下反而不合适？

这一步很像复盘里的核心结论：

**面试官真正做的，不是听你复述事实，而是验证你有没有完成技术判断。**

### 6. 重点准备两类开放题

这类题最能拉开层次：

1. **系统升级题**  
   例如：如何升级成企业知识库？如何支持多租户？如何支持权限？
2. **系统诊断题**  
   例如：效果差了怎么定位？延迟高了先优化哪里？fallback 多了怎么分析？

这两类题，答得好，通常比背定义更能体现工程成熟度。

### 7. 最终备考顺序建议

可以按下面顺序准备：

1. 先写好 1 分钟版和 5 分钟版项目介绍
2. 再为每个模块补“三问卡片”
3. 再准备失败案例
4. 再准备评测与验证逻辑
5. 最后做 mock，练被打断、被追问时的回答稳定性

一句话总结：

**项目面试准备的本质，不是把系统讲一遍，而是把每个设计决策都讲成一个可以被审问也站得住的判断。**
